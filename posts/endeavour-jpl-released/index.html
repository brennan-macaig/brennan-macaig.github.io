Endeavour and JPL released to the public

February 19, 2021 3-minute read

Hey Folks!

Happy to announce that Endeavour and JPL are now both available on my GitHub publicly!

Endeavour and JPL are tools that I am happy to contribute to the open source community, and I’ll take some space here in my blog to explain what they do, why I made them, and how they work.

Endeavour

Endeavour (the last NASA-made thing to bring humans to space) is a tool which effectively mimics cURL, writing files to the raw repository format in Nexus. Initially, endeavour was created to replace a tool written in bash.

At present, endeavour only supports the raw repository format, although I have plans of extending it to support other repository formats.

Endeavour was designed with a few, very simple goals in mind:

  • Be easy to use in a CI/CD pipeline (GitLab’s CI/CD was what was designed for, which is very similar to Travis-CI)
  • Be secure, handling credentials well and not leaking them to a process table or other location
  • Easy to use, with minimal arguments to do the right thing
  • Execute fast
  • No external dependencies other than Nexus being online

With these goals in mind, I’m fairly sure endeavour meets them.

Endeavour can upload files and directories, as well as can upload as many files from as many locations in a single command. This is accomplished entirely with Go’s standard library, without any external dependencies in the supply chain to build.

Although, endeavour is not complete - it would certainly appreciate some work! If you’d like to contribute to endeavour, feel free!

Endeavour can be found on my GitHub

JPL

JPL (the Jet Propulsion Lab) is a tool I created to build software “with rocket speed”. JPL is an alternative to something like Make/Gradle (most similar though to Make). It runs a set of commands in a YAML file line by line, and just captures their output. It doesn’t make crappy assumptions about commands, instead basing actions on their exit codes.

You may be saying “jeez, that sounds a lot like bash”. It is! Except JPL supports templating, and isn’t impossible to read.

JPL files are designed to look very similar to those for GitLab CI/CD / Travis-CI so that people familiar with those tools can basically use JPL out-of-the-box in a basic manner, and can learn more about templating to use it in a more complex way.

JPL was created because

  1. Gradle requires Java (?!) which sucks
  2. In order to understand Make you need to sacrifice your first-born child and sign a contract with the devil in blood
  3. Bash sucks to maintain

JPL can always use more features (modules) to run, as well as needs the following improvements:

  1. The environment passed to commands is probably too wide. I’d like to model what is passed off of sudo.
  2. JPL should support arbitrary “steps” as opposed to only build and test. Invoking jpl run should run the stages in either a user-defined order, or only run build if no order is found and there is a build stage. Otherwise invoking jpl run <stagename> should simply invoke only that stage.
  3. I’d like JPL to be able to easily integrate with HashiCorp Vault.

Other news

I’m very happy to release both endeavour and JPL, and am happy to review any merge requests that you may submit!

If you end up using JPL/endeavour, I look forward to hearing from you about your experiences!
